# 팩토리 패턴

객체 지향 디자인 패턴의 기본 원칙에는 **확장에 있어서는 열려 있어야 하며, 수정에 있어서는 닫혀 있어야 한다** 라는 말이 있다. (OCP)

수정에 닫혀 있어야한다는 말은 코드를 수정하지 않고, 모듈의 기능을 확장하거나 변경할 수 있어야 한다는 의미이다. 그렇기에, **수정이 일어날 가능성이 큰 부분과 그렇지 않은 부분을 분리하는 것**이 좋을것이다.

객체는 변수와 함수가 수정되거나 추가될 가능성이 매우 높다. 그러므로 객체의 생성을 담당하는 코드의 변경 가능성은 매우 높다는 의미이다. 그러므로 객체의 생성을 담당하는 코드들을 한 곳에서 모아 관리한다면 결합도를 줄일 수 있지 않을까? 라는 생각에 팩토리 패턴이 나타났다.

  
> 📌 결합도: 한 클래스에 변경점이 얼마나 다른 클래스에 영향을 주는가를 의미


```java
Class Product {
	public product() {}
}

Class UserA {
	Product p = new Product();
}

Class UserB {
	Product p = new Product();
}

Class UserC {
	Product p = new Product();
}
```

해당 코드에서 Product 클래스가 사라지거나 생성자가 변경된다면 User 클래스들도 다 변경될 것이다. 하지만 만약 Product 객체를 생성하는 코드가 팩토리 패턴으로 한 곳에서 모아서 관리했었더라면? 그 부분만 수정했으면 될 것이다.

이러한 팩토리 패턴을 이용하는 구체적인 방식에는 **팩토리 메서드 패턴**과 **추상 팩토리 패턴**이 있다.

---

## 팩토리 메서드

단일 객체 생성을 상위 클래스가 하지 않고, 하위 클래스에서 결정하도록 위임하는 패턴이다.

간단하게 예시를 하나 들어보자.

![factory-method.png](https://github.com/user-attachments/assets/5344f95e-133b-42a8-8ea0-596b91ac46ae)
)

`Car` 라는 상위 인터페이스가 있고, 자동차의 종류에는 `Suv` , `Sedan` , `Truck` 3가지 종류가 있다. 그리고 각 자동차는 현대 자동차와 기아 자동차들만 있으며, `Suv` 에는 `Santafe`, `Sorento` \ `Sedan` 에는 `K5` , `Sonata` \  `Truck` 에는 `Bongo` , `Porter` 가 있다.

그리고 각 팩토리 클래스를 통해 한가지 종류의 자동차를 생성할 수 있다. 이처럼 `SuvFactory` , `SedanFactory` , `TruckFactory` 같은 하위 팩토리가 해당 자동차 계열의 구체적인 자동차(Santafe, K5 등)를 생성하도록 **객체 생성 책임을 위임**한 것이 **팩토리 메서드 패턴**이다.

### 문제점

이때 팩토리 메서드 중 일부를 살펴보면

![factory-method-logic.png](https://github.com/user-attachments/assets/620dd109-33ea-48af-b088-dcae88e3567b)

다음과 같이 메서드 내에서 분기문을 통해 객체를 생성해주고 있다. 만약 여기서 쌍용 자동차가 추가된다면 어떻게 될까? 모든 팩토리 클래스에다가 쌍용 자동차에 대한 분기문을 추가해주어야 할 것이다. 그렇다면 OCP 원칙에 위배되는 꼴이다. 이러한 부분이 팩토리 메서드의 문제점이라고 할 수 있다.

## 추상 팩토리

서로 관련된 객체들을 생성하는 여러 개의 팩토리 메서드를 하나의 인터페이스로 묶은 것이다.

앞선 예제를 추상 팩토리로 표현하면 다음과 같다.

![abstract-factory.png](https://github.com/user-attachments/assets/7305465c-781d-4715-9552-048f70b61ef3)

팩토리 메서드는 **하나의 종류의 자동차(SUV, Sedan 등)**에 대해서만 객체를 생성하지만,

추상 팩토리는 **한 팩토리 객체를 통해 여러 종류의 자동차(같은 브랜드의 SUV, Sedan, Truck 등)**를 일관되게 생성할 수 있도록 구성된다.

또한, 팩토리 메서드에서는 **어떤 자동차를 만들지 조건문 분기(if/else)**로 처리했다면, 추상 팩토리에서는

**어떤 팩토리 객체를 사용하느냐에 따라 같은 메서드를 호출해도 다른 객체가 생성**된다.

![abstract-factory-logic.png](https://github.com/user-attachments/assets/aaf5a927-f2e1-46ad-abeb-05b371dfab84)


그렇다고 해서 추상 팩토리가 팩토리 메서드의 상위 개념이거나 무조건 더 나은 패턴이라는 뜻은 아니다.

예제와 같이 `여러 제품을 묶어 일관되게 생성해야 하는 상황(제품군 개념)`에서는 추상 팩토리가 유지보수성과 확장성 측면에서 유리할 뿐이다.

예를 들어, 쌍용 자동차가 새로 추가된다고 가정해보자. 

이 경우 기존 코드를 수정할 필요 없이, **쌍용 자동차의 각 차종 클래스들과 전용 팩토리 클래스만 추가**하면 된다. 조건 분기를 수정하는 방식보다 훨씬 OCP(개방-폐쇄 원칙)를 잘 지킨 구조라고 볼 수 있다.

### 문제점

물론 추상 팩토리 패턴이 모든 확장에 대해 유연한 것은 아니다. 예를 들어, 새로운 자동차 종류인 `경차(CompactCar)`가 추가된다면, 모든 팩토리 클래스마다 **`createCompactCar()` 메서드**를 추가해야 한다.

반면, 팩토리 메서드 패턴에서는 단순히 **새 추상 클래스와 구현 클래스만 추가**하면 되므로 더 간단할 수 있다. 따라서 어떤 패턴이 더 우월하다고 단정 짓는 것은 무의미하다. **요구사항의 성격에 따라 적절한 패턴을 선택하거나 조합하는 것이 중요하다.** 두 패턴의 장점을 잘 이해하고, 상황에 따라 유연하게 활용하는 것이 객체지향 설계의 핵심이다.

---

## 결론

객체의 생성을 팩토리 클래스에 위임하여 결합도를 감소시키고, 객체 생성 로직을 캡슐화하여 유지보수가 향상된다는 점에서는 2개의 패턴이 똑같다고 할 수 있다.

그렇다면 대체 차이가 무엇일까?

두 패턴의 차이는 **의도**에 있다. 

**객체 하나의 생성 책임을 하위 클래스에 위임**하고자 하는것이 `팩토리 메서드 패턴`이고, **관련된 객체 묶음 (제품군)을 일관성있게 생성**하는것이 `추상 팩토리 패턴`이다.
